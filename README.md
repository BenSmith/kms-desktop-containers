# KMS Desktop Container

A containerized Wayland desktop (labwc) that drives your monitor directly via
KMS/DRM — no host compositor, display manager, or Wayland/X installation needed.
Bring up a full desktop on any machine with a GPU, monitor, keyboard, and mouse.

The container runs labwc (a wlroots compositor) with systemd as PID 1, waybar,
foot terminal, Firefox, PipeWire audio, and Xwayland for X11 app compatibility.
Everything is baked into the image — no host desktop stack required.

## Quick start

```bash
cd labwc

# One-time host setup
sudo systemctl enable --now seatd
sudo usermod -aG seat,input,video,render $USER
# Log out and back in for groups to take effect

# Generate .env (detects your UID/GID and device group IDs)
./setup.sh

# Build and start
podman compose up -d
```

Switch to the desktop with **Ctrl+Alt+F2** (or another free VT).

## Usage

### compose

```bash
podman compose up -d              # start (builds image on first run)
podman compose down               # stop and remove
podman compose up -d --build      # rebuild image and start
podman exec -it --user $(id -u) desktop /bin/bash   # shell inside
```

### run.sh

```bash
./run.sh [name]          # build, create, and start (default name: desktop)
./run.sh --shell [name]  # open a shell inside
./run.sh --stop [name]   # stop
./run.sh --rm [name]     # remove
podman start [name]      # restart a stopped container
```

Both launchers produce the same container from the same image.

## KVM vs bare metal

The image is the same either way. Three variables tune the compositor for
KVM/virtio-gpu guests:

| Variable | KVM value | Why |
|---|---|---|
| `WLR_RENDERER` | `pixman` | Avoids virgl fence round-trips (~27× faster than gles2) |
| `WLR_NO_HARDWARE_CURSORS` | `1` | virtio-gpu has no hardware cursor plane |
| `WLR_DRM_NO_MODIFIERS` | `1` | virtio-gpu has limited DRM modifier support |

These are set by default. **For bare metal**, remove them:

- **compose**: delete the three `WLR_*` lines from the `environment` section of `compose.yaml`
- **run.sh**: clear the `kvm_env` array: `local kvm_env=()`

wlroots will auto-select the best renderer (Vulkan or GLES2) for the real GPU.

### Venus Vulkan (KVM only)

Venus is a Vulkan-over-virtio-gpu protocol that gives the guest direct access
to the host GPU via the RADV driver. It requires changes to the libvirt/QEMU
configuration and works independently of which wlroots renderer labwc uses.

Enabling Venus requires disabling the QEMU seccomp sandbox
(`seccomp_sandbox = 0` in `/etc/libvirt/qemu.conf`) because libvirt's default
`spawn=deny` policy blocks virglrenderer from forking the render server process
that Venus needs. If you're running `WLR_RENDERER=pixman` and not using native
Wayland Vulkan apps, Venus provides no visible benefit — consider leaving the
sandbox enabled.

See [ACCELERATION.md](ACCELERATION.md) for the full story: setup instructions,
what works, what doesn't, and how to revert Venus if you don't need it.

## .env

`setup.sh` generates this automatically. To write it by hand:

```
DESKTOP_USER=ben
DESKTOP_UID=1000
DESKTOP_GID=1000
HOST_VIDEO_GID=39
HOST_RENDER_GID=105
HOST_INPUT_GID=104
HOST_SEAT_GID=993
```

```bash
id -u                              # DESKTOP_UID
id -g                              # DESKTOP_GID
getent group video  | cut -d: -f3  # HOST_VIDEO_GID
getent group render | cut -d: -f3  # HOST_RENDER_GID
getent group input  | cut -d: -f3  # HOST_INPUT_GID
getent group seat   | cut -d: -f3  # HOST_SEAT_GID
```

## Persistent home

### compose

Stored in a podman named volume (`desktop-home`). Survives container recreates.

```bash
podman volume inspect desktop-home
podman volume rm desktop-home
```

### run.sh

Stored at `~/.local/share/desktop-kms/<name>/` on the host.

## How it works

### Boot sequence

1. **`container-bootstrap.service`** (oneshot, first boot only) — creates the
   container user matching your host UID/GID, sets up supplementary groups,
   sudo, and shell prompt. Guarded by
   `ConditionPathExists=!/etc/container-bootstrapped` so it only runs once.

2. **`labwc.service`** — `ExecStartPre` creates and chowns `/run/user/$UID`
   (the tmpfs is fresh each boot), then `setpriv --reuid --regid --keep-groups`
   drops to your user and launches the compositor under `dbus-run-session`.

### User namespace and device access

The container uses rootless podman with a dual UID mapping:

```
Default rootless:  container UID 0    → host UID 1000  (systemd runs as root inside)
Added via +uidmap: container UID 1000 → host UID 1000  (desktop user owns its files)
```

This means systemd (PID 1) runs as UID 0 inside the container — normal init
behavior — while the desktop user (UID 1000) maps to the same host UID, so
bind-mounted home directory files have correct ownership in both contexts.

`--group-add=keep-groups` preserves host supplementary GIDs (video, render,
input, seat) on PID 1. The labwc service uses `setpriv --keep-groups` to drop
to your user without calling `setgroups()`, which would wipe those GIDs. This
is why the service uses `setpriv` rather than systemd's `User=` directive.

### Audio

Audio runs entirely inside the container — no host PipeWire/PulseAudio needed.
The autostart script detects the ALSA analog output card via `aplay -l`, writes
a PipeWire sink config, and starts the PipeWire stack. This bypasses SPA's udev
enumeration, which fails in containers (podman's `/dev` tmpfs doesn't match the
host udev database).

### Environment passing

Host values (UID/GID/group IDs) are passed as container environment variables.
Systemd services read them via `PassEnvironment=` — no intermediate file needed.

## GPU support

| GPU | Status |
|---|---|
| AMD (GCN / RDNA) | Works out of the box |
| Intel (Haswell+, Arc) | Works out of the box |
| NVIDIA (Nouveau) | Basic 3D, no reclocking — limited performance |
| NVIDIA (proprietary) | Not supported — requires a separate image with the NVIDIA driver |

## Capabilities

All capabilities are dropped, then only the needed ones added back:

| Capability | Why |
|---|---|
| `CHOWN` | Bootstrap: chown home and runtime dir to desktop user |
| `DAC_OVERRIDE` | Bootstrap: write `/etc/sudoers.d`, `/etc/passwd` as root |
| `FOWNER` | Bootstrap: set ownership of files not owned by the process |
| `SETUID` | Bootstrap: `useradd`; compositor: `setpriv --reuid` |
| `SETGID` | Bootstrap: `groupadd`; compositor: `setpriv --regid` |
| `FSETID` | systemd: preserve setuid/setgid bits during file operations |
| `KILL` | systemd: send signals to managed services |
| `NET_BIND_SERVICE` | systemd: bind to privileged ports if needed |
| `SETFCAP` | systemd: set file capabilities |
| `SETPCAP` | systemd: modify process capabilities |
| `SYS_CHROOT` | systemd: `chroot()` for service isolation |
| `SYS_NICE` | Compositor: scheduling priority for smooth rendering |

## Known limitations

**Native Wayland Vulkan clients** — with the default `WLR_RENDERER=pixman`,
the compositor can't display windows from apps that render via Vulkan into
DMA-BUF surfaces (e.g. `vkcube --wsi wayland`). XWayland apps and native
Wayland OpenGL apps work fine. See ACCELERATION.md for details.

**SPA/udev audio enumeration** — PipeWire's SPA ALSA plugin can't enumerate
sound cards via udev inside the container. The workaround is baked in, but only
the first detected analog output is configured. Multiple sound cards may need
manual adjustment to the autostart script.

**Input device hot-plug** — devices plugged in after the container starts won't
be visible. Restart the container to pick up new devices.

**Single monitor** — only tested with one monitor. Multi-monitor may need
config adjustments in `rc.xml`.

**No host PipeWire forwarding** — the container runs its own audio stack driving
ALSA directly. If a host desktop session exists on another VT, they will
conflict over the ALSA device.

**NVIDIA proprietary** — not tested. Requires a separate image with the driver.
